# 执行过程 UI 修复计划

## 背景

执行过程与回答依据已经拆分，但仍有两个细节会误导用户：

1. 工具过程只展示成功读取数量，跳过的不可读网页只在侧栏内提示，摘要容易让用户误以为过程不完整。
2. 思考正文中的裸 URL 后接中文说明时，Markdown 自动链接可能把“，原因是...”一起吞进链接。

## 用例矩阵

| 编号 | 场景 | 输入状态 | 预期 |
| --- | --- | --- | --- |
| UI-A | 搜索 + 读取部分跳过 | `web_search` 成功，`url_read` degraded | 摘要展示 `搜索 1 次 · 跳过 1 个网页`，侧栏不暴露内部服务名 |
| UI-B | 读取成功 + 跳过 | `url_read` success/degraded/failed | 摘要展示成功读取和跳过数量 |
| UI-C | 历史 digest-only | `toolDigests` 包含成功读取和 degraded 读取 | 历史页也展示跳过数量，不出现“未使用”告警 |
| UI-D | 裸 URL 后接中文说明 | `https://example.com/a，原因是...` | 链接 href 只到 URL，中文说明保留 |

## 实施

1. 调整 `executionProcessModel` 的摘要和可渲染条件，把跳过网页作为过程事实展示。
2. `ExecutionProcess` 侧栏在只有跳过记录时也能给出可读说明。
3. 增加 Markdown 裸 URL 预处理工具，并在思考正文和最终回答渲染中共用。
4. 用组件测试覆盖摘要、侧栏脱敏和 URL 边界。
