# Fusion UI/UX 真实交互录制与性能归因报告

## 录制范围

- 时间：2026-06-24 07:24 CST
- 环境：线上 `https://fusion.seanfield.org`，已登录 Chrome 会话
- 页面：Fusion Web 桌面端
- 约束：未启动本地 Fusion dev server，未改代码
- 对应 spec：`docs/superpowers/specs/2026-06-24-ui-ux-performance-polish.md`

本轮录制覆盖：

1. 从新建页切换到已有会话。
2. 从已有会话点击新建对话。
3. 从新建首页点击示例问题进入新会话。
4. 在已有会话中发送普通问题。
5. 在已有会话中发送需要联网搜索的问题。
6. 展开最后一轮回答的思考过程。
7. 展开最后一轮联网回答的搜索来源。

## 采集方法

每条路径采样以下状态：

- URL
- DOM 节点数量
- 页面文本长度
- 首页标题是否可见
- examples 数量
- history hydration skeleton 数量
- `role="status"` 数量
- 输入框是否聚焦
- 回答依据/搜索/思考相关文案是否可见
- 主要滚动容器的 `bottomDelta`
- 控制台 error

采样点：

- 操作前
- 操作后 0ms
- 150ms
- 300ms
- 800ms
- 1500ms
- 对发送类路径额外采样 3000ms、6000ms、10000ms、15000ms

## 采集限制

Chrome 插件的页面沙箱未暴露 `performance`，并且页面 `window` / `document.body` 是不可扩展对象，不能挂载临时 `PerformanceObserver`。本机 Chrome 也未发现可用 CDP 调试端口。

因此本轮无法直接读取：

- `longtask` performance entries
- `layout-shift` performance entries
- 精确资源请求 timing
- React render 次数

本报告中的“重复 render / 布局抖动”只能基于 DOM 节点数量、文本长度、滚动位置和可见状态变化做近似归因。若后续要验证精确 long task / CLS / render 次数，需要新增临时前端 instrumentation 或使用带 CDP tracing 的 Chrome 录制。

## 总体结论

本轮没有发现旧会话正文闪现，也没有发现控制台 error。页面体感卡顿主要不再来自旧内容残留，而更像是以下两类：

1. **会话数据等待感**：切换到未缓存会话时，点击后约 1.5s 才进入目标路由并显示 skeleton，约 2.5s 消息稳定。
2. **发送后流式增长导致的滚动位置波动**：普通发送和联网搜索期间，DOM 和文本持续增长，滚动容器 `bottomDelta` 在多次采样中波动，需要更精确定位消息滚动容器和自动滚动策略。

新建对话路径表现最好：点击后首个采样已经进入新建页，首页标题、examples 和输入框焦点都立即正确。

## 后续优先级建议

### P1-A：会话卡片点击后的即时主区反馈

证据：

- 切换已有会话时 0-800ms 仍停留首页，1.5s 才进入 skeleton，2.5s 才稳定。

建议：

- 点击会话卡片后立即显示目标会话 pending 主区，而不是等路由/数据阶段完成。
- 或至少让侧边栏 active 和主区加载状态同步到点击瞬间。

### P1-B：示例问题点击后的草稿会话进入速度

证据：

- 点击示例问题后 0-800ms 仍在首页，1.5s 才进入会话。

建议：

- 继续压缩 `onDraftCreated` 前的等待。
- 如果后端创建草稿需要时间，前端可以先显示本地 draft shell。

### P1-C：消息滚动容器与 bottom lock

证据：

- 普通发送和联网搜索时 `bottomDelta` 多次波动。
- 当前采样可能抓到滚动容器，但仍需代码级确认。

建议：

- 明确消息列表滚动容器 ref。
- 只在用户位于底部附近时自动跟随。
- 在流式增长、推荐问题出现、资料层出现时保持底部锁定。

### P1-D：思考展开的锚点保持与可观测状态

证据：

- 展开思考过程后 DOM 数量不变，但 bottomDelta 从 51 增至 208。

建议：

- 展开/收起时保持消息锚点。
- 给折叠控件补 `aria-expanded` 和稳定可测状态。

### P2：联网回答阶段状态降噪

证据：

- 联网回答期间 `statusCount` 在 0/1 间多次变化。
- DOM 增量主要来自工具过程、正文、资料层和推荐问题。

建议：

- 保持资料层和工具过程低权重。
- 避免阶段状态卡片在回答完成后继续抢正文视觉层级。

## 结论

当前最值得继续优化的不是“旧内容闪现”，而是：

1. 会话切换和示例发送前 800-1500ms 的感知等待。
2. 流式生成期间的底部跟随和滚动稳定性。
3. 思考过程展开后的锚点保持。

不建议此时做大规模视觉重构。下一步更适合按 P1-A / P1-B / P1-C / P1-D 分别写 implementation plan，再小批次实施。
