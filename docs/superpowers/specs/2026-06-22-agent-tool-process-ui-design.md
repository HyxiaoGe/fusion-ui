# Agent 工具过程区聚合 UI 设计

## 背景

Web 聊天状态主线已经把 assistant 消息里的思考、工具活动、正文输出和推荐问题拆清楚。当前新的问题集中在 Agent 工具过程区：同一步里有多个 `web_search` 或 `url_read` 时，界面会渲染多个带图标的 `搜索`、`读取` chip，看起来像一排可点击按钮；后面紧跟 query、URL、标题和结果数，整体更像调试日志，而不是用户可快速理解的过程说明。

这份设计只处理 Web 端聊天页里的 Agent 工具过程展示。它不改变工具调用本身，也不改变搜索来源、URL 卡片或消息流协议。

## 目标

- 把同类工具调用聚合成少量摘要，例如 `搜索 2 次 · 共 10 条结果`、`读取 2 个网页`。
- 降低成功工具过程的视觉权重，让它解释“刚才做了什么”，但不抢正文、来源和推荐问题的阅读路径。
- 保留失败、降级、运行中状态的可见性，避免用户误以为所有联网步骤都成功。
- 去掉普通成功场景里“多个图标按钮”的观感，只在真正可交互的控件上使用按钮样式。
- 不重复展示完整搜索结果和 URL 内容；过程区负责过程摘要，资料区负责内容材料。

## 非目标

- 不改后端、SSE、Redux store shape、Dexie schema、工具调用协议或落库结构。
- 不重写 `SourcesPanel`、`SourcesSidebar`、`UrlCard` 的资料展示语义。
- 不处理移动端、Electron 或触摸交互。
- 不引入新的整体设计系统。
- 不新增工具结果排序、引用评分、来源可信度判断。
- 不通过启动本地 Fusion 服务验证；验证只使用代码检查、单测、构建和 CI/CD。

## 当前问题

### 1. 同类工具重复渲染成 chip

`ToolCallChip` 目前按 tool call 逐个渲染。一次联网问题可能产生多个搜索 query 或多个 URL 读取目标，于是同一行出现多个 `搜索` 或 `读取` chip。虽然这些 chip 不是主要操作按钮，但视觉上接近按钮，用户会误判它们是否可点、是否代表多个独立动作。

### 2. 过程详情太接近工程日志

`ToolCallSummary` 把 input 到 result 的信息直接拼成行，例如 query、URL、标题、结果数一起出现。信息真实，但没有用户层级：用户需要先知道“搜索了几次、读了几个网页、有没有失败”，而不是先读每个工具调用的完整参数。

### 3. 成功态和异常态视觉层级不够区分

成功工具调用、运行中工具调用、失败工具调用都在同一块结构里出现。普通成功态应低权重，失败、降级、运行中才需要更强提示。

### 4. 过程区和资料区职责有重叠趋势

搜索结果标题、URL 读取标题、来源列表都可能在过程区出现。过程区如果展示太多内容，会和 `SourcesPanel`、`UrlCard` 重复，导致一轮回答里材料出现两遍。

## 设计原则

### 原则 1：聚合优先，详情按需

默认先展示聚合摘要。只有用户需要排查、工具仍在运行、存在失败/降级，或调用数量较多时，才提供可展开详情。

### 原则 2：状态优先于参数

过程区第一眼回答三个问题：

- 调用了什么类型的工具。
- 调用了多少次或处理了多少个目标。
- 是否仍在运行、失败、降级或被截断。

query、URL、标题、结果条数属于第二层详情。

### 原则 3：成功态降权，异常态可见

全部成功完成的工具组使用低饱和、低边框权重。失败、降级、部分失败、运行中状态用明确的状态文案和图标提示，但不使用大面积高亮。

### 原则 4：只把可操作控件做成按钮

普通工具标签是状态标识，不使用按钮式边框和 hover 反馈。展开详情、打开来源、打开 URL 才是按钮。

### 原则 5：资料展示归资料区

过程区可以展示精简标题或 hostname，但不展示完整搜索结果列表。完整来源继续交给 `SourcesPanel` / `SourcesSidebar`，URL 内容继续交给 `UrlCard`。

## 信息架构

### 聚合单位

以 `AgentStepCard` 内的 tool calls 为输入，按工具类型和状态聚合成 `ToolCallGroup`：

```ts
type ToolCallGroupKind = "web_search" | "url_read" | "other";

type ToolCallGroupStatus =
  | "running"
  | "success"
  | "partial"
  | "degraded"
  | "failed";
```

聚合规则：

- `web_search`：按所有搜索调用聚成一个搜索组。
- `url_read`：按所有网页读取调用聚成一个读取组。
- 其他工具：按工具名聚合；未知工具归入 `other`。
- 如果同组里同时有成功和失败，状态为 `partial`。
- 如果存在 running 调用，状态优先显示 `running`。
- 如果没有成功结果且有失败，状态为 `failed`。

### 摘要文案

`web_search`：

- 运行中：`正在搜索 · 2 个查询`
- 成功：`搜索 2 次 · 共 10 条结果`
- 部分失败：`搜索 2 次 · 1 次失败`
- 失败：`搜索失败 · 2 个查询`
- 降级：`搜索降级 · 已跳过外部结果`

`url_read`：

- 运行中：`正在读取网页 · 2 个目标`
- 成功：`读取 2 个网页`
- 部分失败：`读取 2 个网页 · 1 个失败`
- 失败：`网页读取失败 · 2 个目标`
- 降级：`网页读取降级 · 已跳过部分页面`

其他工具：

- 运行中：`正在调用工具 · 2 个任务`
- 成功：`调用 2 个工具`
- 部分失败：`调用 2 个工具 · 1 个失败`
- 失败：`工具调用失败 · 2 个任务`

### 详情展示

默认折叠规则：

- 全部成功且调用数小于等于 2：默认只展示摘要。
- 全部成功且调用数大于 2：展示摘要和 `查看详情`。
- running、partial、degraded、failed：展示摘要，并默认展开一层短详情。

详情内容：

- 搜索详情展示 query 和结果数，每行最多一条 query，不显示完整结果标题列表。
- URL 详情展示 hostname 或页面标题，避免长 URL 撑开布局。
- 失败详情展示失败目标和短错误文案。
- 超长 query、标题、URL 使用单行截断；完整值通过 `title` 或可访问 label 保留。

## 交互设计

### 聚合行

每个工具组展示为一条紧凑状态行：

- 左侧：低权重图标，运行中使用 spinner。
- 中间：摘要文案。
- 右侧：仅当有详情时展示展开按钮。

聚合行不是按钮，除非点击区域明确绑定展开行为。展开按钮需要有可访问名称，例如 `查看搜索详情`。

### 展开详情

展开后以列表形式显示短详情：

- 每行左侧不重复工具图标。
- 每行只保留一个主信息和一个辅助信息。
- 详情列表不使用卡片套卡片，避免在 `AgentStepCard` 里继续堆叠框。

### 异常提示

失败、降级、部分失败状态在摘要行里直接可见。详情里展示最多 3 条异常目标，超过时显示 `还有 N 个目标未展示`。

### 与来源和 URL 卡片的关系

过程区不提供“打开来源侧栏”的主入口。用户要看来源，仍从正文附近的来源组件进入。过程区只解释工具过程，避免同一个来源入口出现两套。

## 组件设计

### 新增工具分组 helper

建议新增：

- `src/lib/agent/toolCallGroups.ts`

职责：

- 接收 `AgentToolCall[]`。
- 输出聚合后的 `ToolCallGroup[]`。
- 生成摘要字段、状态字段和详情字段。
- 不读取 React 状态，不访问 DOM，不发请求。

这样可以用纯函数测试覆盖分组逻辑，避免把聚合规则散落在 TSX 里。

### `AgentStepCard`

调整职责：

- 保留 step 外壳、标题、状态和展开行为。
- 把原来的逐个 tool call chip 区域替换为工具组列表。
- 对异常 step 保持明显可见。
- 对普通完成 step 低权重展示，避免压过回答正文。

### `ToolCallSummary`

调整为聚合详情渲染组件：

- 输入从单个 tool call 改为 `ToolCallGroup` 或 group detail。
- 负责 query、hostname、结果数、错误短文案的紧凑展示。
- 不再输出完整 `input -> output` 日志风格文本。

### `ToolCallChip`

保留为内部状态标识或删除普通成功路径里的使用：

- 如果保留，应改成不可点击的 badge 视觉。
- 不能在同一组里重复渲染多个相同工具名 chip。
- 运行中 spinner 可以由聚合行承载，不依赖 chip。

## 视觉规则

- 正常成功组：中性色文字、浅边界或无边界、无按钮 hover。
- running 组：保留 spinner 和状态文案，颜色与当前状态主线保持一致。
- failed / degraded / partial：使用现有语义色 token，不新增硬编码主题色。
- 文本密度比当前略高，但行高不能低于可读水平。
- 一行摘要必须在桌面宽度下稳定不换成多行按钮堆叠；细节内容允许截断。

## 可访问性

- 展开按钮使用真实 `button`，包含 `aria-expanded`。
- 图标若只是装饰，设置为 `aria-hidden`。
- 运行中状态通过文字表达，不只依赖 spinner。
- 失败、降级文案必须可被屏幕阅读器读到。
- 长 URL 截断后，完整 URL 放在 `title` 或等效可访问文本中。

## 测试计划

### 单元测试

新增或更新 helper 测试：

- 多个 `web_search` 聚合成一个搜索组。
- 多个 `url_read` 聚合成一个读取组。
- 搜索结果数正确求和。
- 成功和失败混合时状态为 `partial`。
- running 调用存在时组状态为 `running`。
- 未知工具归入 `other` 或按工具名聚合。

### 组件测试

更新 `AgentStepCard` / `ToolCallSummary` 相关测试：

- 两个搜索调用只渲染一条 `搜索 2 次` 摘要，不再出现两个按钮式 `搜索`。
- 两个 URL 读取调用只渲染一条 `读取 2 个网页` 摘要。
- 失败或部分失败时，失败文案默认可见。
- 成功详情默认折叠，点击展开后显示 query 或 hostname。
- 运行中工具组显示 spinner 和运行中文案。
- 长 query / URL 不撑开容器。

### 回归测试

- 现有聊天状态主线测试继续通过。
- `npm test` 通过。
- `npm run build` 通过。
- `git diff --check` 通过。

## 实施边界

- 第一阶段只改前端展示和测试。
- 不修改后端 tool result schema。
- 不改消息持久化结构。
- 不启动本地开发服务器。
- 不顺手清理历史未使用组件，除非实现过程中证明它们已经被新路径完全替代且删除不扩大风险。

## 验收标准

- 用户在联网问题里不会再看到同一步里重复堆叠多个 `搜索` / `读取` 图标按钮。
- 工具过程区能一眼看出搜索几次、读取几个网页、是否失败或降级。
- 正文、来源、URL 卡片和推荐问题的阅读路径不被过程详情抢占。
- 异常工具调用不会被折叠到完全不可见。
- 不依赖本地服务验证，提交后可通过单测、构建和 CI/CD 确认。
