# Fusion UI/UX 性能与交互优化 Spec

## 背景

Fusion Web 端已经完成一轮围绕会话切换、新建对话、首页示例问题、输入框焦点、会话 hydration、推荐问题状态的优化。当前体感明显更流畅，但这些优化仍然主要来自局部问题修正。

下一阶段需要把“继续打磨”从零散小改动收敛为可验收、可分批、可回归测试的优化计划。重点不再是大改视觉风格，而是围绕真实高频路径继续降低感知卡顿、旧内容闪现、状态跳动和交互反馈不一致。

本 spec 只覆盖 Web 端桌面体验，不考虑移动端。

## 目标

1. 让会话切换、新建对话、首次进入会话、发送消息、联网搜索后的阅读路径保持稳定、顺滑。
2. 降低用户可感知的旧内容闪现、loading 抖动、按钮状态滞后和布局跳动。
3. 用明确验收标准约束每批优化，避免继续堆零散视觉小修。
4. 保持现有架构边界：不重写 Redis Stream、SSE、消息持久化、推荐问题生成逻辑。
5. 所有实现批次都必须能通过单测、构建、CI/CD 和线上 Chrome 验收闭环。

## 非目标

- 不做移动端适配。
- 不重做整体视觉品牌或设计系统。
- 不重构后端对话流协议。
- 不改变联网搜索、URL 读取、回答依据的数据结构契约。
- 不启动本地 Fusion dev server 作为验收依据。
- 不把性能问题泛化为大型状态管理重写，除非后续真实指标证明必要。

## 当前已完成基线

### 会话切换与新建对话

- 从已有会话点击新建对话时，立即显示首页 surface，不再短暂保留旧会话正文。
- 首页 examples 不再阻塞首屏输入框出现。
- 首次进入首页/新建对话后，输入框自动聚焦。
- 新建对话按钮有 active/pressed 状态。

### 消息区与资料区

- 回答依据已收敛为资料层，默认只显示来源摘要和有限卡片，详情通过展开查看。
- 工具过程、回答依据、正文、推荐问题的视觉层级比早期更清晰。
- 会话 hydration 使用轻量 skeleton，避免大段“正在加载”文案抢占注意力。

### 推荐问题

- 推荐问题已随 assistant 消息持久化，刷新后不重新生成。
- 推荐问题 loading、换一批、点击发送中状态已有基础反馈。

## 优先级

### P0：真实交互性能归因

先建立一组固定的 Web 端验收路径，用 Chrome 线上环境观察真实行为，而不是继续凭体感改代码。

覆盖路径：

1. 从已有会话切换到另一个已有会话。
2. 从已有会话点击新建对话。
3. 从新建首页点击示例问题进入草稿会话。
4. 在已有会话里发送普通问题。
5. 在已有会话里发送需要联网搜索的问题。
6. 展开/收起思考过程、回答依据、搜索来源详情。

验收标准：

- 切换到未缓存会话时，不显示上一段会话正文。
- 新建对话后 800ms 内首页标题、示例问题和输入框可见。
- 首页输入框在新建对话后自动聚焦，且不抢已有会话页面焦点。
- 关键路径控制台无新增 error。
- 记录每条路径是否存在肉眼可见卡顿、旧内容闪现、loading 闪烁、布局跳动。

建议产物：

- 一份短记录：每条路径的现象、疑似原因、是否需要代码改动。
- 若发现长任务或重复渲染，再进入 P1/P2 实施。

### P1：会话切换与消息区稳定性

目标是继续打磨“切换会话”这条最高频路径。

候选优化：

- 区分首次 hydration skeleton、已有缓存复用、流式恢复三种状态。
- 保证未加载会话不会短暂显示旧会话消息。
- 已加载过的会话切换时减少 layout shift。
- 对消息列表较长的会话，保持直接定位到最新回复。

验收标准：

- 首次进入未缓存会话时，只显示 skeleton 或空白占位，不显示旧会话正文。
- 已缓存会话切换时不出现明显白屏。
- 进入会话后默认位于最新回复区域。
- `ChatMessageList` 相关单测覆盖 `history-hydration`、空会话、已有消息三类状态。

### P1：首页与新建对话工作流

目标是让新建对话成为稳定、轻量、可立即输入的工作台。

候选优化：

- 示例问题轮换继续保持低打扰：hover、focus、点击前暂停。
- 新建按钮状态与当前路由保持一致。
- examples API 慢或失败时不影响输入框出现。
- 首页输入框聚焦只在首页/新建场景触发，不影响已有会话。

验收标准：

- examples 未返回时 fallback 示例立即显示。
- hover 示例超过一个轮换周期，当前文案不变化。
- 点击新建对话后，旧会话正文不残留。
- 输入框自动聚焦，并且 `activeElement.id === "chat-message-input"`。

### P1：输入框与推荐问题状态统一

目标是减少“按钮能不能点、正在做什么”的不确定感。

候选优化：

- 推荐问题生成、换一批、点击发送中统一为低权重状态。
- 输入框上传、文件失败、模型不可用、发送中、停止生成保持尺寸稳定。
- 发送按钮和停止按钮不造成工具栏布局跳动。
- 推荐问题点击后避免重复发送。

验收标准：

- 推荐问题 loading 使用 `role="status"`。
- 换一批和发送中使用 `aria-busy` 或等价状态标记。
- 推荐问题 pending 期间重复点击不触发第二次发送。
- `ChatInput.test.tsx` 和 `SuggestedQuestions.test.tsx` 覆盖主要状态。

### P2：侧边栏高频操作细节

目标是提升会话列表长期使用时的稳定感。

候选优化：

- active 状态、hover 操作、长标题截断保持一致。
- 重命名、删除、生成标题入口更清晰，但不增加视觉噪音。
- 搜索框 loading、空结果、错误状态更轻量。
- 分组日期与会话卡片间距继续收紧。

验收标准：

- active 会话在列表刷新后仍可见。
- 搜索中不触发分页加载。
- 搜索错误不遮挡列表操作。
- 长标题不会挤压操作按钮。

### P2：回答资料层与工具过程继续收敛

目标是让联网回答更容易扫读，减少工具过程对正文的干扰。

候选优化：

- 回答依据默认显示来源摘要，展开后显示全部来源详情。
- 搜索、读取 URL、工具过程保持同一资料层语言。
- 工具过程在回答完成后默认降低视觉权重。
- 正文引用入口与资料层互相对应，避免“搜索 5 条但只看到 3 张卡片”的困惑。

验收标准：

- 回答依据摘要数量与展开详情数量语义一致。
- 卡片数量限制有明确文案，例如“显示 3 条，查看全部 5 条”。
- 工具过程完成后不抢正文视觉层级。
- 联网回答刷新前后资料层显示一致。

## 实施原则

1. 先归因，再改代码。性能或卡顿相关问题必须先说明是网络、数据等待、bundle 加载、React 渲染、滚动定位还是视觉反馈问题。
2. 小批次提交。每批只解决一类交互问题，避免一次提交混合多个不相关改动。
3. TDD 优先。能写单测的状态先写失败测试，再实现。
4. Subagent-Driven 默认用于适合拆分的实施批次，主 agent 保持协调和审查。
5. 不用本地 Fusion dev server 验收。验证优先级是代码检查、非启动测试、构建、CI/CD、线上 Chrome 已登录环境。
6. 不为了视觉 polish 引入新状态源；优先沿用现有 hooks、Redux slice 和组件边界。

## 测试策略

每个实施批次至少包含以下之一：

- 组件单测：覆盖具体 UI 状态、交互和可访问属性。
- hook 单测：覆盖派生状态、缓存、去重、pending 状态。
- 页面级测试：覆盖路由切换、fallback、hydration 这类跨组件行为。

推荐命令：

```bash
npm test -- --run <受影响测试文件>
npm test
npm run build
```

说明：

- 当前项目没有独立 `lint` script。
- `next build` 当前会跳过 lint 和类型校验；如果后续补独立校验命令，应写入对应 implementation plan。

## Chrome 验收清单

每批部署后，在已登录的线上 Fusion Web 页面执行：

1. 强刷页面，确认加载后无控制台 error。
2. 从已有会话点击新建对话，确认旧正文不残留。
3. 新建页面确认输入框聚焦、示例问题可见。
4. 点击一个示例问题，确认进入草稿/新会话路径正常。
5. 切换两个已有会话，确认不显示上一段对话内容。
6. 对联网回答展开回答依据和工具过程，确认视觉层级与数量文案一致。

## 风险与边界

- 真实线上数据和登录态会影响验收结果，因此不能用本地启动替代线上 Chrome 验收。
- 会话切换优化容易影响流式恢复、草稿会话和历史会话三条路径，实施时必须分别覆盖。
- 输入框自动聚焦不能影响已有会话的阅读、选择文本和侧边栏搜索。
- 推荐问题 pending 状态不能吞掉发送失败后的恢复能力。
- 回答依据数量文案必须基于同一份来源数据，避免摘要和详情不一致。

## 后续拆分建议

第一批建议先做 P0 归因，不改或少改代码：

1. 建立 Chrome 线上验收记录模板。
2. 记录六条关键路径的实际表现。
3. 按“明显卡顿/轻微不顺/无需处理”分级。
4. 再根据记录决定 P1 的具体 implementation plan。

第二批再做 P1 中最高收益的单点：

1. 会话切换与 hydration 稳定性。
2. 首页新建对话工作流。
3. 输入框与推荐问题状态统一。
