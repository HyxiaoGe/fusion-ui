# Runtime Config Readability Design

## 背景

运行时配置入口已经能完成查看、校验、创建候选版本、激活和禁用，但当前页面直接展示 `namespace/key/version/payload` 等工程字段。管理员进入页面时很难判断每项配置影响什么、是否能改、改错会影响哪条用户路径。

## 目标

- 让管理员第一眼看到配置的中文名称、类别、影响范围和风险提示。
- 保留内部标识，但降级为辅助信息，便于排查和与后端记录对齐。
- 默认不铺开原始 JSON，只展示字段摘要；需要编辑时仍保留 JSON payload。
- 不改后端接口，不引入数据库迁移，不改变配置结构。

## 方案

前端新增一层配置展示元数据，将已落库的配置项映射到产品语义：

- `prompt_template/*` 归类为 Prompt 模板，说明具体影响标题、推荐问题、工具边界或文件处理。
- `agent_strategy/default` 归类为 Agent 策略，说明会影响搜索、深读、来源排序和工具上下文。
- `model_presentation/default` 归类为模型展示，说明会影响模型选择器里的能力说明。
- 未登记配置使用保守 fallback，明确提示“尚未登记用途说明”。

payload 摘要器只做确定性展示：

- 字符串显示字段名和字数。
- 数字、布尔、数组、对象分别显示数量或状态。
- 超过 4 个字段时显示剩余字段数量。

## UI 行为

- 当前生效配置：主标题使用中文名称，副标题说明影响范围；内部标识、来源、版本、当前值摘要放在卡片内。
- 创建候选版本：根据当前 namespace/key 展示“正在编辑什么配置”和风险提示。
- 版本列表：主标题使用中文名称，版本和内部标识作为辅助信息；保留激活/禁用动作。

## 测试

- 纯函数测试覆盖已知配置元数据、fallback 和 payload 摘要。
- 组件测试覆盖当前生效配置中文说明、表单配置说明和版本列表中文名称。
- 保留既有创建、校验、激活、禁用行为测试。
